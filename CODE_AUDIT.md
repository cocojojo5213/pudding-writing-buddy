# Code Audit

This file records the current hardening pass after reading the project source with line numbers. The main objective was to find defects that could lose manuscript data, hide persistence failures, or make the writing assistant unreliable during longer novel workflows.

## Current Pass Summary

- `lib.js` lines 18 and 140-168: `DEFAULT_PROJECT` is now deep-frozen, `normalizeProject()` preserves an existing `updatedAt`, rounds legacy fractional `targetWords` values to whole counts, and the project schema carries a `versionToken` used for write-conflict detection.
- `lib.js` normalization helpers: legacy imported characters/hooks/outline/timeline/resources/arcs/chapters that lack ids now receive deterministic content-based fallback ids instead of fresh random ids on every read, and duplicate or control-character legacy collection ids are repaired with stable replacement ids so frontend id-based editing cannot target the wrong record. Missing chapter `createdAt` no longer gets a new timestamp during pure normalization, explicit empty chapter `body` values are no longer overwritten by legacy `text` fields, and matching user entries no longer inherit default-story character knowledge, hook notes, resources, or arc details just because they share a demo name or text.
- `lib.js` exported offline helpers now normalize sparse/null project objects and treat non-object payloads as empty payloads before reading nested fields. Direct calls to `buildPrompt()`, `offlineAssist()`, `offlinePlan()`, `offlineDraft()`, `offlineAudit()`, `offlineBrainstorm()`, and settlement helpers no longer crash on missing project collections, null payloads, or non-string draft/style/revision text; empty settlement input still returns the intended "without body text" validation error.
- `lib.js` chapter reference, settlement report, and export helpers: numeric-looking chapter ids are now resolved as ids before being treated as chapter numbers, and empty or fractional chapter refs fall back to the next chapter instead of producing `第1.5章` or wrapping prior context to the last chapter. Chapter summaries, scalar metadata lines, and generated settlement report fields now use single-line sanitation, so summary, genre, logline, title, timeline, or ledger update text cannot inject fake Markdown sections while manuscript body text remains intact.
- `lib.js` lines 566-657: chapter settlement now inserts a chapter when the client sends an unsaved chapter, merges partial incoming chapter payloads with the saved chapter before extracting summary/hooks/character knowledge, preserves existing body/plan/audit fields, updates truth ledgers, records character knowledge from evidence sentences instead of only generic text, appends character/hook/resource ledger notes instead of overwriting older state, refreshes the prior auto-generated settlement timeline event for the same chapter instead of accumulating stale summaries, keeps same-title chapter settlement timeline entries separate by `chapterId`, and preserves timeline entries that were manually revised after settlement.
- `server.js` lines 63-157: API method guards now return `405` with an `Allow` header for known routes. Invalid JSON, non-object JSON request bodies, unknown assist tasks, and malformed explicit assist `payload`/`modelConfig` shapes are covered by regression tests.
- `server.js` static serving: malformed percent-encoded paths now return a plain `400 Bad request`; static files only allow `GET`/`HEAD`; encoded slash traversal attempts are covered by regression tests; `HEAD` returns headers without a body; URL parsing uses a fixed localhost base instead of trusting the request `Host` header; and successful static responses read the file before sending `200` headers, so both `GET` and `HEAD` read failures return a clean `404` instead of reporting inconsistent success.
- `server.js` request parsing: JSON bodies are now assembled as buffers before UTF-8 decoding, so multibyte characters split across network chunks are not corrupted. Oversized request bodies are rejected before retaining the chunk that crosses the size limit. Top-level JSON must be an object.
- `server.js` startup config: `PORT` and `STALE_TEMP_MS` are parsed as bounded integer settings with safe defaults, so malformed, fractional, zero, or out-of-range environment values no longer crash startup or distort stale temp cleanup.
- `server.js` lines 189-287: corrupt `project.json` is backed up before resetting to a default project, and corrupt-backup filenames include a random suffix to avoid overwrite collisions. Missing files still bootstrap cleanly. Saves are atomic, return the exact project written to disk, tolerate stale-temp cleanup races, and rotate `versionToken` on every write.
- `server.js` lines 204-256: write routes now serialize `versionToken` checks and disk writes through one in-process queue. Concurrent same-version saves cannot both pass the check; stale project, settle, and reset writes return `409` instead of overwriting newer work. Blind writes without any version marker are also rejected when a project already exists, and malformed version metadata such as array/object tokens or timestamps is rejected before it can be coerced into a matching string. `/api/project` now requires a complete project payload with scalar project fields, core ledger arrays, object-shaped collection items, nonblank/unpadded/control-character-free unique ids for collection items, required string fields for each ledger item, valid project language, valid hook statuses, and decimal whole-number target word counts inside the same 300-20000 range used by core normalization, so bare tokens, sparse project objects, malformed fields, invalid enum values, unsafe ids, duplicate ids, fractional/exponent/hex/out-of-range targets, or malformed collections cannot normalize into defaults/bounds and overwrite the manuscript. Routes that accept an optional explicit project now apply the same complete-shape checks instead of silently falling back to disk/default state. `/api/settle` now uses the current disk project as its write base after the version check, so stale client project payloads cannot overwrite saved title, notes, or chapter metadata during settlement. Malformed explicit settlement chapter payloads are rejected before any disk write.

## Residual Risks

- The settlement extractor is heuristic. It is useful for offline state capture, but it does not fully understand entity aliases, implied character knowledge, or resource quantities.
- Offline drafting remains deterministic and template-shaped. It is serviceable as a no-model fallback, but model-backed drafting is still the intended higher-quality path.
- Write-conflict protection is in-process. It protects the local single-process server used by this app; if the app is later deployed as multiple Node processes writing the same file, the same version check would need an inter-process lock or database transaction.
- Frontend behavior still has only focused DOM-stub regression coverage plus browser screenshots/manual runtime probes, not a full browser automation suite.
